Task 2: Database Design using Entity Modelling

Entity-Relationship (ER) modelling is a high-level conceptual data model diagram. ER diagrams are used to represent the overall logical structure of a database graphically. They are a visual way to describe the data that will be stored in a database and the relationships between pieces of data.

Entity Relationship Components:

  • Entities: Represent real-world objects or concepts (e.g., Student, Course, Patient). They are typically represented as rectangles.
  • Attributes: Properties or characteristics of an entity (e.g., Student Name, Course Code, Patient Age). They are represented as ovals connected to entities.
  • Relationships: Associations between two or more entities (e.g., a Student *enrolls in* a Course, a Patient *suffers from* an Ailment). They are represented as diamonds.
  • Cardinality: Defines the number of instances of one entity that can be associated with the number of instances of another entity (e.g., One-to-One, One-to-Many, Many-to-Many).

Implementing Relationships in Tables:

Relationships are implemented in relational databases primarily through the use of Primary Keys (PK) and Foreign Keys (FK).

  • Primary Key (PK): A unique identifier for each record in a table.
  • Foreign Key (FK): A field in one table that uniquely identifies a row of another table. The foreign key is defined in a second table, but it refers to the primary key in the first table. It establishes a link between the two tables.

Example (from Patient Database):


CREATE TABLE pDetails (
    patNo INT PRIMARY KEY,
    name VARCHAR(255),
    gender CHAR(1),
    residence VARCHAR(255),
    age INT
);

CREATE TABLE pAilments (
    patNo INT,
    ailment VARCHAR(255),
    status VARCHAR(255),
    FOREIGN KEY (patNo) REFERENCES pDetails(patNo)
);

In this example, `patNo` in `pDetails` is the Primary Key. `patNo` in `pAilments` is a Foreign Key, linking each ailment record to a specific patient in `pDetails`.

Task 3 (Conceptual): Normalization Rules

Database normalization is the process of organizing the columns and tables of a relational database to minimize data redundancy and improve data integrity. It involves breaking down a large table into smaller, less redundant tables and defining relationships between them.

Key Normalization Forms:

  • First Normal Form (1NF):
    • Each column must contain atomic (indivisible) values.
    • There are no repeating groups of columns.
  • Second Normal Form (2NF):
    • Must be in 1NF.
    • All non-key attributes must be fully functionally dependent on the primary key (no partial dependencies).
  • Third Normal Form (3NF):
    • Must be in 2NF.
    • No transitive dependencies (non-key attributes should not be dependent on other non-key attributes).
  • Boyce-Codd Normal Form (BCNF):
    • A stricter version of 3NF. For every non-trivial functional dependency X -> Y, X must be a superkey.

Task 6: Apply Right Join and Left Join to Database Tables

SQL JOIN clauses are used to combine rows from two or more tables, based on a related column between them.

Types of Joins:

  • INNER JOIN: Returns records that have matching values in both tables.
  • LEFT (OUTER) JOIN: Returns all records from the left table, and the matching records from the right table. If there is no match, the result is NULL from the right side.
  • RIGHT (OUTER) JOIN: Returns all records from the right table, and the matching records from the left table. If there is no match, the result is NULL from the left side.
  • FULL (OUTER) JOIN: Returns all records when there is a match in either left or right table records.

Practical Implementation (using 'games' and 'duties' tables):

Assuming 'games' and 'duties' tables are populated as in Task 4, and 'Code' is the linking column.

Left Join example:


USE sports;
SELECT g.Name, g.Game, d.title
FROM games g
LEFT JOIN duties d ON g.Code = d.Code;

Right Join example:


USE sports;
SELECT g.Name, g.Game, d.title
FROM games g
RIGHT JOIN duties d ON g.Code = d.Code;

Inner Join example (most common):


USE sports;
SELECT g.Name, g.Game, d.title
FROM games g
INNER JOIN duties d ON g.Code = d.Code;

Task 8: Apply OOP to Databases

Applying Object-Oriented Programming (OOP) concepts to databases involves managing data in a way that aligns with the principles of object-oriented languages (encapsulation, inheritance, polymorphism).

Object-Oriented Databases (OODBs) / Object Databases (ODBs):

  • Concept: Databases designed to store and manage data as objects directly. They inherently support OOP principles.
  • Implementation: Data is stored as instances of classes, complete with their attributes and methods. Relationships between data are handled via object references.
  • Manipulation: Objects are directly manipulated in the database, reflecting their state and behavior as defined in the class.

Object-Relational Mapping (ORM):

The more common approach for relational databases. An ORM provides a bridge between object-oriented programming languages and relational databases.

  • Mapping: Translates objects in your programming language (e.g., a `Patient` class) to rows in database tables (e.g., `pDetails` table) and vice-versa.
  • Abstraction: Allows developers to interact with database data using object-oriented code, abstracting away raw SQL queries.
  • Examples: Entity Framework (.NET), Hibernate (Java), SQLAlchemy (Python).

Task 9: Apply Inheritance in Databases

Inheritance in databases, particularly in the context of OOP, refers to how a hierarchy of classes (where one class inherits properties from another) is represented and managed within a database system.

Inheritance in Object-Oriented Databases (OODBs):

  • OODBs natively support inheritance. A subclass object automatically inherits the attributes and methods of its superclass.
  • This direct mapping simplifies the representation of complex hierarchies.

Inheritance with Object-Relational Mapping (ORM):

When using an ORM with a relational database, inheritance is typically handled using strategies:

  • Table Per Hierarchy (TPH): All classes in an inheritance hierarchy are mapped to a single table. A 'discriminator' column indicates the specific type of object stored in each row. 
    
-- Example for TPH: Person and subclasses Student, Employee
CREATE TABLE Persons (
    Id INT PRIMARY KEY,
    Name VARCHAR(255),
    Discriminator VARCHAR(50), -- 'Student' or 'Employee'
    EnrollmentDate DATE,      -- Student specific
    HireDate DATE             -- Employee specific
);
  • Table Per Type (TPT): Each class in the inheritance hierarchy (including abstract base classes and derived classes) has its own table. Relationships between these tables are established using primary key/foreign key constraints. 
    
-- Example for TPT: Person, Student, Employee
CREATE TABLE Persons (
    Id INT PRIMARY KEY,
    Name VARCHAR(255)
);

CREATE TABLE Students (
    Id INT PRIMARY KEY,
    EnrollmentDate DATE,
    FOREIGN KEY (Id) REFERENCES Persons(Id)
);

CREATE TABLE Employees (
    Id INT PRIMARY KEY,
    HireDate DATE,
    FOREIGN KEY (Id) REFERENCES Persons(Id)
);
  • Table Per Concrete Class (TPC): Each concrete (non-abstract) class in the hierarchy is mapped to its own table. This table contains all attributes of the class, including inherited ones, leading to some data duplication but simpler queries for specific types.

Task 10: Create and manipulate views tables

A view is a virtual table based on the result-set of an SQL statement. A view contains rows and columns, just like a real table. The fields in a view are fields from one or more real tables in the database.

Practical Implementation:

Create view 'AllGames':


USE sports;
CREATE VIEW AllGames AS SELECT * FROM games;

Extract all records from 'AllGames' view:


USE sports;
SELECT * FROM AllGames;

Extract details for Code 3 from the two tables (using views implicitly):


USE sports;
SELECT * FROM games WHERE Code = 3;
SELECT * FROM duties WHERE Code = 3;

Extract records of athletics or basketball from games folder (using views implicitly):


USE sports;
SELECT * FROM games WHERE Game = 'athletics' OR Game = 'basketball';

Rename a view (Example: Renaming 'AllGames' to 'CurrentGames'):


-- SQL Server specific syntax for renaming a view
USE sports;
EXEC sp_rename 'AllGames', 'CurrentGames';

Drop a view (Example: Dropping 'CurrentGames' view):


USE sports;
DROP VIEW CurrentGames;
